## 3. Reliability Patterns

### Circuit Breaker

Prevent cascading failures by failing fast.

```java
public class CircuitBreaker {
    private State state = State.CLOSED;
    private int failureCount = 0;
    private long lastFailureTime;

    public Response call(Request request) {
        if (state == State.OPEN) {
            if (shouldAttemptReset()) {
                state = State.HALF_OPEN;
            } else {
                throw new CircuitOpenException();
            }
        }

        try {
            Response response = doCall(request);
            onSuccess();
            return response;
        } catch (Exception e) {
            onFailure();
            throw e;
        }
    }

    private void onSuccess() {
        failureCount = 0;
        state = State.CLOSED;
    }

    private void onFailure() {
        failureCount++;
        lastFailureTime = System.currentTimeMillis();
        if (failureCount >= threshold) {
            state = State.OPEN;
        }
    }
}
```

```java
public <T> T retryWithBackoff(Callable<T> operation) {
    int attempts = 0;
    while (attempts < maxRetries) {
        try {
            return operation.call();
        } catch (RetryableException e) {
            attempts++;
            if (attempts >= maxRetries) throw e;

            long delay = (long) Math.pow(2, attempts) * baseDelay;
            delay += random.nextLong(delay / 2); // jitter
            Thread.sleep(Math.min(delay, maxDelay));
        }
    }
}
```

## 5. API Design Patterns

### API Versioning

```
URI Versioning:
GET /api/v1/users
GET /api/v2/users

Header Versioning:
GET /api/users
Accept: application/vnd.api+json;version=2

Query Parameter:
GET /api/users?version=2
```

### Pagination

```
Offset-based (traditional):
GET /api/users?offset=20&limit=10
Response: { data: [...], total: 1000 }

Problem: Inconsistent results if data changes during pagination

Cursor-based (recommended):
GET /api/users?cursor=abc123&limit=10
Response: {
  data: [...],
  next_cursor: "def456",
  has_more: true
}

Benefits:
- Consistent results
- Efficient for large datasets
- Works well with real-time data
```

### Idempotency

```
Problem:
Client в”Җв”Җв”Җ POST /payments в”Җв”Җв”Җв–¶ Server
       в—Җв”Җв”Җ (network timeout) в”Җв”Җв”Җ
       в”Җв”Җв”Җ POST /payments в”Җв”Җв”Җв–¶ (duplicate payment!)

Solution: Idempotency Key
Client в”Җв”Җв”Җ POST /payments в”Җв”Җв”Җв–¶ Server
           Idempotency-Key: abc123
       в—Җв”Җв”Җ 200 OK в”Җв”Җв”Җ
       в”Җв”Җв”Җ POST /payments в”Җв”Җв”Җв–¶ Server recognizes key
           Idempotency-Key: abc123
       в—Җв”Җв”Җ 200 OK (cached response, no duplicate)

Implementation:
1. Client generates unique key per operation
2. Server stores: key вҶ’ response
3. If key exists, return stored response
4. If new key, process and store response
```

## Quick Reference: Pattern Selection

| Problem | Pattern |
|---------|---------|
| Services need to communicate | API Gateway, Message Queue |
| Need distributed transactions | Saga (Choreography/Orchestration) |
| Read-heavy workload | CQRS, Read Replicas |
| Need audit trail | Event Sourcing |
| Prevent cascading failures | Circuit Breaker, Bulkhead |
| Handle transient failures | Retry with Backoff |
| Safe deployments | Blue-Green, Canary |
| Gradual feature rollout | Feature Flags |
| Debug distributed systems | Distributed Tracing |
