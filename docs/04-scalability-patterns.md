### Making Services Stateless

**Before (Stateful):**
```java
// Session stored in server memory
HttpSession session = request.getSession();
session.setAttribute("user", user);
```

**After (Stateless):**
```java
// State in external store
String sessionId = request.getHeader("Session-Id");
User user = redisClient.get("session:" + sessionId);

// Or use JWT
String token = request.getHeader("Authorization");
User user = jwtService.verify(token);
```

### Sharding Strategies

#### Hash-Based Sharding
```
shard_id = hash(partition_key) % num_shards

Pros: Even distribution
Cons: Resharding is expensive, no range queries
```

#### Range-Based Sharding
```
Shard 0: A-H
Shard 1: I-P
Shard 2: Q-Z

Pros: Range queries possible
Cons: Hot spots (e.g., many users with names starting with 'S')
```

### Data Locality Strategies

| Strategy | Description | Trade-off |
|----------|-------------|-----------|
| **Single Region** | All data in one region | Simple but high latency |
| **Read Replicas** | Replicas in each region | Fast reads, slow writes |
| **Multi-Primary** | Primary in each region | Complex conflict resolution |
| **Partitioned** | Data owned by region | Data sovereignty friendly |

### Batch Processing

```
Without batching:
INSERT INTO users VALUES (1, 'A');  -- 1 round trip
INSERT INTO users VALUES (2, 'B');  -- 1 round trip
INSERT INTO users VALUES (3, 'C');  -- 1 round trip
Total: 3 round trips

With batching:
INSERT INTO users VALUES
  (1, 'A'),
  (2, 'B'),
  (3, 'C');  -- 1 round trip
Total: 1 round trip
```

### Pagination

```
Offset-based (problematic at scale):
SELECT * FROM posts ORDER BY id LIMIT 20 OFFSET 10000;
-- Must scan 10,020 rows to return 20

Cursor-based (efficient):
SELECT * FROM posts
WHERE id > :last_seen_id
ORDER BY id
LIMIT 20;
-- Uses index, always fast
```

### Retry with Exponential Backoff

```
Attempt 1: Failed вҶ’ Wait 1s
Attempt 2: Failed вҶ’ Wait 2s
Attempt 3: Failed вҶ’ Wait 4s
Attempt 4: Failed вҶ’ Wait 8s
Attempt 5: Failed вҶ’ Give up

With jitter (prevent thundering herd):
wait_time = min(cap, base * 2^attempt) + random_jitter
```

## Quick Reference: When to Use What

| Problem | Solution |
|---------|----------|
| Too many requests | X-axis scaling (add servers) |
| Database reads slow | Read replicas, caching |
| Database writes slow | Sharding, async writes |
| Service too complex | Y-axis (microservices) |
| Data too large | Z-axis (sharding) |
| High latency for distant users | CDN, multi-region |
| Traffic spikes | Auto-scaling, message queues |
| Service failures cascade | Circuit breaker, bulkhead |
| Too many retries at once | Exponential backoff with jitter |
